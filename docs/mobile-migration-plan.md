# Mobile Migration Plan

This document is the execution plan for migrating StickToGif from a browser-first Vite app to iOS and Android apps using Capacitor.

It is intentionally specific to this repository. It is not generic Capacitor advice.

## Goals

Ship a mobile app that can:

1. Open reliably on iOS and Android.
2. Load the bundled sample clip.
3. Import GIF, MP4, and MOV files from the device.
4. Let the user place a target and run tracking.
5. Apply sticker, text, or blur overlays.
6. Export the result as GIF and, where supported, animated WebP.
7. Share or save the result through native mobile flows.
8. Keep all processing local on-device.

## Non-Goals For V1

These should not block the first mobile release:

1. Cloud sync.
2. User accounts.
3. Background processing after the app is suspended.
4. Gallery management inside the app.
5. Complex camera capture workflows.
6. Tablets or foldables as a separate UX target.

## Public Repository Constraint

This repository is public. That means the mobile migration must be designed so that no confidential material is committed to the repo.

This is a hard requirement, not a best-effort guideline.

### Nothing confidential may be stored in the repository

The repo must never contain:

1. iOS signing certificates.
2. iOS provisioning profiles.
3. Android upload keystores.
4. Keystore passwords.
5. Key aliases or alias passwords.
6. Apple Developer team identifiers if they are considered private for this project.
7. App Store Connect API keys.
8. Google Play service account keys.
9. CI secrets of any kind.
10. Private backend credentials if a backend is added later.
11. Any `.p12`, `.mobileprovision`, `.jks`, `.keystore`, `.json`, `.env`, or similar secret-bearing files tied to release credentials.

### The repo must remain fully public-safe

A public clone of the repository should be able to:

1. Install dependencies.
2. Build the web app.
3. Generate the Capacitor projects.
4. Run the app locally in simulator or emulator development mode.

A public clone of the repository should not be able to:

1. Produce a store-signable release build without external credentials.
2. Access any private Apple or Google publishing account.
3. Reveal team ownership, signing identity details, or release secrets that are not meant to be public.

### Configuration must be split into public and private layers

Public layer:

1. Source code.
2. Public docs.
3. Capacitor config that is safe to expose.
4. Build scripts that work without secrets for local development.
5. Example templates with placeholder values only.

Private layer:

1. Signing identities.
2. Keystores and certificates.
3. Provisioning profiles.
4. Store publishing credentials.
5. CI/CD secrets.
6. Any team-specific configuration that should not be exposed publicly.

### Required policy for mobile signing

The repository should support two completely different concerns:

1. Public development builds.
2. Private release signing.

Public development builds:

1. Must work without private keys.
2. May use simulator builds or debug builds generated by local developer tooling.
3. Must not require any checked-in secret material.

Private release signing:

1. Must happen outside the public repository.
2. Must use environment-injected secrets or machine-local secure storage.
3. Must never depend on committed secret files.

### Required repo hygiene

Tasks:

1. Expand `.gitignore` to exclude all likely mobile secret files.
2. Add a `docs/mobile-secrets-policy.md` document describing what must never be committed.
3. Add a checked-in example environment or config template only if it contains placeholders, never real values.
4. Add a pre-commit or CI check to fail if known signing file types appear in the repo.
5. Review generated iOS and Android project files for team identifiers or signing references before committing them.

Suggested `.gitignore` additions:

1. `*.jks`
2. `*.keystore`
3. `*.p12`
4. `*.mobileprovision`
5. `*.cer`
6. `*.pem`
7. `*.key`
8. `.env`
9. `.env.*`
10. `android/key.properties`
11. Any local signing override files for iOS and Android

### Required release engineering rule

The public repo may contain the native project structure, but it must not contain live release-signing configuration.

That means:

1. Android Gradle files may support reading signing values from ignored local files or environment variables.
2. Xcode project settings may exist, but private team-specific signing configuration should be applied locally or in private CI.
3. Release workflows must be documented, but the actual secrets must be injected privately.

### Documentation requirement

The mobile migration is not complete until the repo contains:

1. A public-safe setup guide for local development builds.
2. A public-safe signing policy document.
3. A release checklist that explicitly separates public steps from private signing steps.

### Definition of done for the public-repo constraint

The mobile repo is compliant only when all of the following are true:

1. No signing or publishing secret is committed.
2. The repo can be cloned and built in development mode by the public.
3. Release signing depends on ignored local files, private CI secrets, or secure external credential storage.
4. The docs clearly state which steps are public and which steps require private credentials.
5. Secret-bearing files are ignored and blocked by validation checks.

## Current Repo Assessment

The repo is a strong candidate for Capacitor because:

1. It is already a frontend-only React + TypeScript + Vite app.
2. The main media pipeline is browser-based and local.
3. Tracking is already offloaded to a worker.
4. The canvas interaction model already supports touch-oriented pointer events.
5. The project builds cleanly and tests pass.

The repo is not yet mobile-ready because:

1. It has no Capacitor setup.
2. It has a hardcoded hosted-web Vite base path in [vite.config.ts](/Users/jaime/workspace/stickToGif/vite.config.ts).
3. It relies on web download semantics in [src/App.tsx](/Users/jaime/workspace/stickToGif/src/App.tsx).
4. It relies on worker boot and dynamic `opencv.js` loading that must be validated in WKWebView and Android WebView.
5. The import UX in [src/components/DropZone.tsx](/Users/jaime/workspace/stickToGif/src/components/DropZone.tsx) is desktop-web flavored.

## Critical Risks

These are the high-risk items that should be validated first.

### 1. Worker + OpenCV bootstrap

Files involved:

1. [src/tracking/trackObject.ts](/Users/jaime/workspace/stickToGif/src/tracking/trackObject.ts)
2. [src/tracking/tracker.worker.ts](/Users/jaime/workspace/stickToGif/src/tracking/tracker.worker.ts)
3. [public/opencv.js](/Users/jaime/workspace/stickToGif/public/opencv.js)

Why it is risky:

1. The worker is created with Vite worker bundling.
2. The worker loads `opencv.js` dynamically using `importScripts()`.
3. It falls back to `fetch()` and `new Function(code)` when `importScripts()` fails.
4. WebViews, especially iOS WKWebView, are stricter and can differ from desktop browsers.

Validation criteria:

1. Worker starts successfully on both platforms.
2. `opencv.js` is resolved from the app bundle correctly.
3. Tracking completes for the sample clip.
4. No silent worker crash or unhandled rejection occurs.

### 2. Video decode in mobile WebViews

Files involved:

1. [src/media/decodeVideo.ts](/Users/jaime/workspace/stickToGif/src/media/decodeVideo.ts)
2. [src/media/decodeSource.ts](/Users/jaime/workspace/stickToGif/src/media/decodeSource.ts)

Why it is risky:

1. Video import relies on `HTMLVideoElement`.
2. Frame extraction relies on repeated seek operations.
3. Frame extraction depends on canvas drawing and `toBlob()`.
4. iPhone-captured MOV files may expose codec and decode edge cases.

Validation criteria:

1. GIF import works.
2. MP4 import works.
3. MOV import works on iPhone-originated files.
4. Sampling frames completes without stalls or crashes.

### 3. Native export flow

Files involved:

1. [src/App.tsx](/Users/jaime/workspace/stickToGif/src/App.tsx)
2. [src/gif/exportGif.ts](/Users/jaime/workspace/stickToGif/src/gif/exportGif.ts)
3. [src/webp/exportWebp.ts](/Users/jaime/workspace/stickToGif/src/webp/exportWebp.ts)

Why it is risky:

1. Current export uses `navigator.share` when available.
2. Fallback export uses a synthetic anchor click.
3. Anchor download behavior is not a reliable mobile-native save strategy.

Validation criteria:

1. User can export and share a generated file on both platforms.
2. Exported files have the expected filename and MIME type.
3. Save/share failures are surfaced clearly.

## Recommended Delivery Strategy

The migration should happen in six milestones:

1. Add Capacitor without changing product behavior.
2. Fix build and asset loading for native WebViews.
3. Prove import and tracking on real devices.
4. Replace web export semantics with native mobile export semantics.
5. Adapt UX for phone-first use.
6. Harden, test, and prepare store-distribution builds.

## Exit Criteria For Mobile Beta

The repo is ready for a closed mobile beta when all of these are true:

1. Capacitor projects are present and reproducible from the repo.
2. Mobile build output resolves assets correctly.
3. The worker loads and tracking runs on both platforms.
4. GIF and MP4 import work on both platforms.
5. MOV support is either working or explicitly rejected with a clear message.
6. Export and native share work on both platforms.
7. The UI is usable on a phone with safe-area support.
8. At least one iPhone and one Android device complete the full flow.
9. No blocker crash remains open.
10. No signing or release secret exists anywhere in the committed repository.
